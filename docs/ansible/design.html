Architecture Overview

The Ansible approach implements Infrastructure-as-Code (IaC) principles for enterprise MSSQL deployment. It provides reproducible, scalable, and maintainable deployment automation.

Key Design Principles

1. Modularity

Each deployment phase is separated into discrete tasks within the mssql role. This allows reuse, testing, and modification without affecting other components.

2. Idempotency

All tasks are designed to be idempotent—running them multiple times produces the same result. This enables safe re-runs and updates without duplicating resources.

3. Variable Hierarchy

Configuration follows a strict hierarchy: CLI args → host vars → group vars → defaults. This enables flexibility while maintaining sensible defaults.

4. Error Handling

Tasks include retry logic for network operations and conditional execution based on host roles. Failed tasks generate detailed logs for troubleshooting.

Role Structure

defaults/main.yml

Default values used when variables aren't overridden:

  • MSSQL version, edition, port
  • Directory paths (backup, data, log)
  • SA password (should override in group_vars)
  • Tools path

tasks/main.yml

Orchestrates all subtasks in order:

  • Import install.yml
  • Import configure.yml
  • Import adventureworks.yml
  • Import backup.yml (conditional for VM1)
  • Import restore.yml (conditional for VM2)

handlers/main.yml

Event handlers for service management:

  • Restart SQL Server service
  • Reload configuration

Variable Management

Hierarchy (Highest to Lowest Priority)

  1. CLI Arguments: ansible-playbook -e var=value
  2. Host Variables: host_vars/vm1.yml
  3. Group Variables: group_vars/mssql_servers.yml
  4. Role Defaults: roles/mssql/defaults/main.yml

Key Variables

  • mssql_version: 2019
  • mssql_edition: Developer or Enterprise
  • sa_password: Strong password (must override)
  • mssql_port: 1433
  • backup_dir: /backup/striped/
  • data_dir: /var/opt/mssql/data/
  • instance_name: instance1 or instance2
  • backup_location: primary or secondary

Inventory Structure

hosts.ini

[mssql_servers]
vm1 ansible_host=192.168.56.101 instance_name=instance1
vm2 ansible_host=192.168.56.102 instance_name=instance2

[mssql_servers:vars]
ansible_user=root
ansible_ssh_private_key_file=~/.ssh/id_rsa

Group Variables (group_vars/mssql_servers.yml)

mssql_version: 2019
sa_password: "YourStrongPassword!@#"
mssql_port: 1433
backup_dir: /backup/striped
data_dir: /var/opt/mssql/data

Host Variables (host_vars/vm1.yml)

instance_name: instance1
backup_location: primary

Conditional Execution

Certain tasks run only on specific hosts:

  • Backup tasks: Only on VM1 (when backup_location == 'primary')
  • Restore tasks: Only on VM2 (when backup_location == 'secondary')

This prevents unnecessary execution and ensures backup/restore operates as intended.

Error Handling

Retry Logic

Network-dependent tasks include retry logic:

  • Retries: 3 times
  • Delay: 10 seconds between retries
  • Applies to: download, SSH operations

Task Conditionals

  • when: Conditional execution based on variables
  • changed_when: Determine if task made changes
  • failed_when: Custom failure conditions

Playbook Execution

site.yml (Main Deployment)

  • pre_tasks: OS validation
  • role invocation: Run mssql role on all hosts
  • post_tasks: Summary report

backup.yml (Backup & Restore)

  • Run backup tasks on VM1 only
  • Run restore tasks on VM2 only

AWX Integration

Configuration files for AWX orchestration (optional):

  • project.yml: Git-based project definition
  • inventory.yml: AWX inventory import
  • credentials.yml: SSH credential template
  • job-template.yml: Job template definitions

GitLab CI/CD Pipeline

Automated pipeline for code quality and deployment:

  • Lint Stage: YAML syntax check, ansible-lint
  • Deploy Stage: Manual trigger for actual deployment

Performance Considerations

  • Parallel Execution: Strategies configured for optimal performance
  • Batch Sizing: Default batch size 1 for safe execution
  • Timeouts: Set appropriately for long operations
  • 10x Backup Stripes: Parallel backup for performance

Security Practices

  • SSH Key Auth: No password authentication
  • Vault: Support for Ansible Vault for sensitive data
  • Credential Management: Secure handling of SA password
  • Audit Logging: Complete execution history

Scalability

The architecture supports scaling to multiple VMs:

  • Add new hosts to inventory/hosts.ini
  • Create host_vars/vm3.yml, etc.
  • Run playbook against all or subset of hosts
  • Conditional execution handles role-based differences

Maintenance & Updates

Future enhancements can be made by:

  • Adding new tasks to appropriate phase
  • Creating new playbooks for additional operations
  • Updating variables without modifying tasks
  • Extending roles for additional functionality

✅ Architecture Complete

The Ansible architecture provides a solid, scalable, and maintainable solution for enterprise MSSQL deployment.
