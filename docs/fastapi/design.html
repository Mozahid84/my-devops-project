📊 Complete Design Document - Detailed architecture for lightweight API deployment

Architecture Overview

The FastAPI approach provides a simple, modern REST API service for MSSQL deployment automation. It leverages FastAPI's async capabilities and integration with Ansible for deployment execution.

Key Design Principles

1. Asynchronous Design

The API uses FastAPI's async/await patterns to handle long-running deployment tasks without blocking. Background tasks execute playbooks while the API remains responsive.

2. Separation of Concerns

Routes are organized by functionality (deploy, health, logs), making the API modular and maintainable. The AnsibleRunner class handles Ansible execution separately.

3. Configuration Management

Configuration uses Pydantic's BaseSettings for environment-based configuration. Supports .env files and environment variables with sensible defaults.

4. Health & Readiness Checks

Comprehensive health checks verify Ansible availability, inventory access, disk space, and system resources. Enables safe deployment in containerized and orchestrated environments.

Core Components

FastAPI Application (main.py)

Purpose: Central FastAPI application with router configuration

  • App initialization with title and version
  • Router registration (deploy, health, logs)
  • Global exception handler
  • Root endpoints for info and version

Configuration (config.py)

Purpose: Centralized configuration management

Setting Type Default
ANSIBLE_INVENTORY Path ../ansible-mssql-deploy/inventory/hosts.ini
ANSIBLE_PLAYBOOK_DIR Path ../ansible-mssql-deploy
MSSQL_SA_PASSWORD String Password123!@#
API_TIMEOUT Integer 3600 (1 hour)

Ansible Runner (ansible_runner.py)

Purpose: Execute Ansible playbooks as subprocesses

  • run_playbook(): Execute playbook with parameters
  • validate_inventory(): Parse and validate inventory
  • ping_hosts(): Test connectivity to all hosts
  • get_inventory(): Return inventory JSON
  • execution_history: Track all executions

API Routes & Endpoints

Deploy Routes (/api/v1/deploy)

Method Endpoint Response Description
POST /install 202 Accepted Full deployment (install, configure, database)
POST /backup 202 Accepted Create backups and restore
POST /install-tools 202 Accepted Install tools only
POST /restore-db 202 Accepted Restore AdventureWorks
GET /status 200 OK Deployment configuration
GET /hosts 200 OK Inventory in JSON
GET /history 200 OK Execution history list
POST /ping 200 OK Test host connectivity

Health Routes (/api/v1/health)

Endpoint Checks Purpose
/check Basic status Simple liveness probe
/ready Ansible, inventory, disk, CPU, memory Readiness probe for deployment
/live Service alive status Kubernetes liveness probe

Log Routes (/api/v1/logs)

Endpoint Parameters Purpose
/latest lines=50 Last N lines of logs
/level/{level} level=INFO|WARNING|ERROR Filter by log level
/since minutes=10 Logs from last N minutes
/clear — Clear log file
/info — Log file statistics

Request/Response Pattern

Background Task Execution

Long-running operations use FastAPI's BackgroundTasks:

  1. Client: Sends POST request to /deploy/install
  2. API: Receives request, queues background task
  3. API: Returns 202 Accepted immediately
  4. Background: Task executes playbook in background
  5. Client: Polls /logs/latest and /deploy/history
  6. Client: Receives status updates

Response Format

{
  "status": "initiated",
  "task_id": "uuid",
  "timestamp": "2026-04-19T10:30:00Z"
}

Data Flow

Deployment Request Flow

  1. HTTP POST: Client sends deployment request
  2. FastAPI: Route handler receives request
  3. Background Tasks: Add task to queue
  4. HTTP 202: Return immediately
  5. Subprocess: Execute ansible-playbook command
  6. Ansible: Run playbook against targets
  7. Logging: Write output to app.log
  8. History: Store execution record

Status Monitoring Flow

  1. HTTP GET: Client polls /logs/latest
  2. FastAPI: Read log file
  3. Response: Last N lines returned
  4. HTTP GET: Client polls /deploy/history
  5. FastAPI: Return execution history
  6. Response: Status and results

Configuration Strategy

Loading Order

  1. Environment variables (highest priority)
  2. .env file (if exists)
  3. Hardcoded defaults (lowest priority)

Key Configuration Variables

  • ANSIBLE_INVENTORY: Path to hosts.ini
  • ANSIBLE_PLAYBOOK_DIR: Directory containing playbooks
  • MSSQL_SA_PASSWORD: SA account password
  • API_TIMEOUT: Execution timeout (seconds)
  • LOG_DIR: Where to store logs

Error Handling

Exception Handling

  • FileNotFoundError: Missing inventory or playbook
  • TimeoutError: Deployment exceeds timeout
  • ConnectionError: SSH connectivity issues
  • ValidationError: Configuration issues

Graceful Degradation

  • Health checks return detailed error messages
  • Logs capture all errors for troubleshooting
  • Execution history preserves failure information

Logging Strategy

Log Levels

  • DEBUG: Detailed diagnostic information
  • INFO: General operation information
  • WARNING: Warnings for potential issues
  • ERROR: Error conditions
  • CRITICAL: Critical errors

Log Output

  • File: ./logs/app.log
  • Format: Timestamp, level, message
  • Rotation: When file exceeds size

Docker Integration

Dockerfile Strategy

  • Python 3.11-slim base image
  • Install Ansible and SSH
  • Copy application code
  • Expose port 8000
  • Health check configured

Docker Compose

  • Single service definition
  • Volume mounts for SSH keys
  • Environment variable passing
  • Health check integration

Testing

Unit Tests (test_api.py)

  • Root endpoint test
  • API version test
  • Health check test
  • Status endpoint test

Running Tests

pytest tests/test_api.py -v
pytest tests/test_api.py -v --cov=app

Security Considerations

Authentication

  • API runs in trusted network (no auth by default)
  • Can be extended with OAuth2 or API key

SSH Key Security

  • SSH keys mounted read-only in Docker
  • Permissions set to 600 (user only)
  • Never logged or exposed

Password Management

  • SA password passed via environment
  • Never logged or exposed in responses
  • Masked in execution output

Performance Optimization

Async Operations

  • Background tasks don't block the API
  • Multiple deployments can queue simultaneously
  • Responsive UI/monitoring possible

Resource Management

  • Disk space checks prevent failures
  • Memory monitoring prevents OOM
  • CPU utilization tracked

Scalability

Horizontal Scaling

  • Multiple instances with shared Ansible inventory
  • Load balancer for API endpoint
  • Shared log storage for centralization

Limitations

  • Single control machine for Ansible
  • Execution history stored locally (use DB for persistence)

Future Enhancements

  • Database-backed execution history
  • Authentication/Authorization
  • Webhook notifications
  • Scheduled deployments
  • Multiple playbook templates
  • Advanced analytics dashboard
